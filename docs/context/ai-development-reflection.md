# AI Development Reflection

## Purpose

이 문서는 프로젝트 코드 변경 로그가 아니라, AI와 함께 이 공모주 일정/분석/알림 서비스를 만들어 가는 방식에 대한 중간 회고다.

이번 회고의 주제는 다음과 같다.

- 현재 프로젝트의 구조와 난이도 평가
- 프론트엔드, 백엔드, 운영 영역의 경계
- AI 코딩에서 사용자의 문제 접근 방식 평가
- 프롬프트를 더 잘 쓰기 위한 기준
- AI가 빠진 전제와 리스크를 질문하게 하는 개인화 설정 방향
- 이 프로젝트를 포트폴리오/면접에서 설명하기 위해 보완해야 할 점

## Current Project Assessment

이 프로젝트는 단순한 화면 구현이나 CRUD 앱보다 난이도가 높다. Next.js App Router 기반의 공개 화면, 관리자 화면, 서버 액션, job API, Prisma/PostgreSQL 모델, 외부 데이터 수집, Vercel Cron, 메일 발송, 운영 로그, 보안 경계가 한 프로젝트 안에 함께 있다.

난이도는 실무 기준으로 `중상~상`에 가깝다. 다만 대규모 분산 시스템, 금융 거래 원장, 의료/보안 규제 시스템 같은 `최상급` 난이도와는 성격이 다르다. 이 프로젝트의 어려움은 대규모 트래픽보다 다음 영역에 있다.

- 외부 데이터 품질과 출처별 신뢰도
- 날짜, 시간대, cron 지연
- 알림 중복 발송 방지
- 공개 화면과 관리자/운영 데이터의 경계
- 점수/추천 정보의 노출 책임
- 장애 시 원인 추적이 가능한 운영 로그
- AI가 만든 코드를 사람이 검증해야 하는 구조

## Frontend And Backend Boundary

프론트엔드 영역은 단순한 마크업보다 넓다. 홈, 상세, 로그인, 관리자 화면뿐 아니라 서버 컴포넌트에서 전달받는 read model의 의미를 이해하고, 공개 화면에 무엇을 보여줄지 판단해야 한다.

프론트엔드가 주로 담당하는 영역은 다음과 같다.

- 홈 캘린더, 종목 카드, 검색/필터/정렬 UX
- 상세 페이지 quick facts, 체크 포인트, 일정 표시
- 로그인 및 관리자 화면 UI
- 반응형 레이아웃과 상태 표시
- 공개 화면에서 admin/source metadata가 노출되지 않도록 하는 표현 경계

백엔드와 더 가까운 영역은 다음과 같다.

- Prisma schema와 DB migration
- OpenDART, KIND, SEIBro, broker source 수집
- daily sync, alert job, Vercel Cron
- 메일 발송, idempotency, stale 처리
- 운영 로그와 job API 인증
- 점수 계산과 score snapshot

이 프로젝트에서 프론트엔드와 백엔드는 완전히 분리된 영역이 아니다. 공개 화면에 어떤 데이터를 내려줄지, 데이터가 비거나 신뢰도가 낮을 때 어떻게 보여줄지, 관리자 전용 정보가 public read path에 섞이지 않게 할지 같은 지점은 반드시 협의가 필요하다.

## Development Approach Review

현재 접근 방식은 전반적으로 실전적이다. 기능을 한 번에 완성하려고 하기보다, 실제 운영 중 생기는 문제를 기준으로 시스템을 보강해 왔다.

좋게 보이는 점은 다음과 같다.

- 문제를 구체적으로 잡는다. 예를 들어 `10시 분석 메일 지연`, `daily-sync transaction 실패`, `public metadata 노출 위험`처럼 실제 운영 이슈에서 출발한다.
- 단순 구현보다 운영 안정성을 함께 본다. 중복 발송, stale 처리, source validation, secret redaction, admin auth 같은 지점이 그 예다.
- AI 코딩 실험 주제로 적절하다. 시간, 데이터 품질, 공개/비공개 경계, idempotency처럼 AI가 흔들리기 쉬운 문제가 포함되어 있다.
- `issue.md`와 `docs/context/`로 결정과 맥락을 남긴다. 이는 AI 작업에서 프로젝트 기억장치 역할을 한다.

아쉬운 점은 접근법 자체의 문제라기보다, 프로젝트가 커지면서 생기는 정리 비용에 가깝다.

- 스레드가 기능별로 나뉘면 AI가 이전 결정의 이유를 매번 복원해야 한다.
- 제품의 1차 정체성이 넓어질 수 있다. 캘린더, 분석, 알림, 관리자, 점수 기능이 모두 중요해 보이면 사용자-facing 핵심 가치가 흐려질 수 있다.
- 점수 기능은 매력적이지만 데이터 신뢰도와 표현 책임이 커서 재오픈 기준이 필요하다.
- 이력서/면접 관점에서는 구현 사실보다 `왜 그렇게 설계했는지`를 설명하는 언어가 아직 더 필요하다.

## Thread Strategy

컨텍스트 한계 때문에 스레드를 나누는 것은 맞는 판단이다. 다만 나누는 기준은 파일이나 기능보다 목표/문제 단위가 더 좋다.

좋은 스레드 단위 예시는 다음과 같다.

- `10시 분석 메일이 믿을 수 있게 나가게 만들기`
- `공개 홈에서 운영 메타데이터가 노출되지 않게 하기`
- `외부 데이터가 깨져도 전체 동기화가 무너지지 않게 하기`
- `관리자가 오늘 운영 상태를 빠르게 판단할 수 있게 하기`
- `점수 기능을 다시 공개할 수 있는 상태로 만들기`

이렇게 목표를 잡으면 한 스레드 안에서 프론트, 백엔드, 테스트, 문서가 함께 움직여도 맥락이 유지된다.

## Prompting Guidance

앞으로 AI에게 작업을 요청할 때는 `무엇을 고쳐줘`보다 `어떤 상태가 되게 해줘`가 더 효과적이다.

좋은 요청 구조는 다음과 같다.

```text
이번 목표:
- 무엇을 달성하려는지

성공 기준:
- 사용자가 어떤 상태를 보게 되는지
- 운영/보안상 지켜야 할 조건
- 테스트나 확인 기준

깨면 안 되는 것:
- 기존 pause/rollout 상태
- 공개 화면 노출 금지 정보
- 알림 중복 방지
- DB migration/cron/env 변경 허용 여부

참고:
- 관련 문서나 파일
```

다만 사용자가 항상 명확한 기준을 줄 수는 없다. 따라서 AI에게 다음 역할을 기대하는 것이 좋다.

- 요청이 모호하면 목표와 성공 기준을 함께 정리한다.
- 작업을 막는 핵심 질문만 1~3개 묻는다.
- 질문이 필요 없으면 합리적인 가정을 명시하고 진행한다.
- 구현 중 놓친 리스크를 짧게 짚는다.

## Suggested Personalization

범용 개인화 설정으로는 다음 문구가 적절하다.

```text
나는 AI와 소프트웨어 프로젝트를 기획, 설계, 구현한다. 내가 항상 명확한 목표나 성공 기준을 제시하지 못할 수 있으므로, 요청이 모호하면 AI가 먼저 문제 정의, 목표, 성공 기준, 제약 조건, 검증 방법을 함께 정리해줘.

작업을 막는 리스크나 빠진 전제가 있으면 핵심 질문 1~3개만 먼저 해줘. 질문이 필요 없으면 합리적인 가정을 명시하고 바로 진행해줘.

기능 구현뿐 아니라 제품 가치, 사용자 경험, 데이터 흐름, 보안/권한, 운영 안정성, 장애 가능성, 테스트 가능성을 함께 고려해줘.

기능 단위보다 목표/문제 단위로 스레드를 구조화하도록 도와줘. 내가 놓친 리스크나 더 나은 접근이 있으면 부드럽게 짚되, 실행 가능한 다음 단계 중심으로 도와줘.
```

이 설정의 목적은 사용자가 모든 기준을 혼자 완벽히 챙기지 못해도, AI가 빠진 전제와 리스크를 함께 보완하도록 만드는 것이다.

## Resume And Interview Readiness

이 프로젝트는 AI와 함께 성과를 만든 포트폴리오로 쓸 수 있다. 다만 이력서/면접에서는 `AI로 만들었다`가 핵심이 아니라, 사용자가 어떤 문제를 정의하고 어떤 판단을 했는지가 핵심이다.

설명할 수 있어야 하는 질문은 다음과 같다.

- 왜 Next.js App Router를 선택했나?
- 외부 데이터가 틀리거나 비었을 때 어떻게 처리했나?
- 알림 중복 발송을 어떻게 막았나?
- Vercel Cron 지연 문제를 어떻게 다뤘나?
- 공개 화면과 관리자 데이터 경계는 어떻게 보장했나?
- AI가 만든 코드의 품질은 어떻게 검증했나?
- 점수 기능을 왜 pause했나?
- 가장 어려웠던 운영 이슈는 무엇이었나?

현재는 구현에 집중한 만큼, 위 질문에 대한 답변 언어가 아직 덜 정리되어 있다. 이는 프로젝트를 이력서에 쓰기 어렵다는 뜻이 아니라, 다음 단계로 회고와 설명 문서를 만들 필요가 있다는 뜻이다.

추천하는 후속 문서는 다음과 같다.

- `docs/context/interview-notes.md`
- 주요 기술 결정별 Q&A
- 각 Q&A에는 `상황`, `선택`, `이유`, `대안`, `결과`, `아쉬운 점`, `근거 파일`을 적는다.

## Next Direction

앞으로 기능을 더 붙이기 전에, 다음 방향을 우선순위로 두는 것이 좋다.

1. 서비스의 1차 정체성을 더 좁힌다.
   - 예: `공모주 청약 마감 전에 오늘 볼 종목과 판단 포인트를 알려주는 서비스`
2. 데이터 품질 상태를 사용자와 관리자 화면에서 더 일관되게 표현한다.
   - 확정값, 미확인, 오래된 값, 출처 불일치 등을 어떻게 보여줄지 정한다.
3. `/admin`을 로그 뷰어보다 운영 콘솔에 가깝게 발전시킨다.
   - 오늘 동기화, 메일 준비, 발송, 제외 사유, stale job, source 실패를 빠르게 판단하게 만든다.
4. 점수 기능은 재오픈 전에 신뢰도, 최신성, 산출 근거, 표현 정책을 먼저 정한다.
5. 이력서/면접용으로 기술 결정 Q&A를 정리한다.

## Summary

현재 문제 접근 방식은 대체로 올바르다. 기능을 만든 뒤 실제 운영 리스크를 보강하는 흐름은 개인 프로젝트와 AI 코딩 실험에 잘 맞는다.

보완해야 할 핵심은 사용자가 목표와 성공 기준을 항상 완벽하게 제시하지 못하는 상황에서, AI가 빠진 전제와 리스크를 먼저 질문하고, 목표/문제 단위로 작업을 구조화하도록 만드는 것이다.

이 프로젝트는 단순히 AI가 코드를 만든 결과물이 아니라, 사용자가 실제 운영 문제를 정의하고 AI를 활용해 풀스택 서비스의 구조와 안정성을 함께 쌓아 간 사례로 정리할 수 있다.
